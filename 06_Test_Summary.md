# Test Summary

## 1. Общая информация

Проект: Web UI тестирование интернет-магазина  
Область проверки: регистрация, вход, каталог товаров, карточка товара, корзина  
Тип проверки: ручное тестирование web-интерфейса  
Формат проверки: первая QA-сессия + первичный тест-дизайн + фиксация дефектов

---

## 2. Цель проверки

Целью данной проверки было:

- понять основной пользовательский путь в системе;
- проверить базовую работоспособность критичных пользовательских сценариев;
- выявить первые дефекты и слабые места;
- зафиксировать начальный набор test cases, checklist и bug reports для портфолио.

---

## 3. Что было проверено

В рамках первой QA-сессии были проверены следующие области:

### 3.1 Регистрация

- успешная регистрация нового пользователя;
- повторная регистрация на уже существующий email;
- обязательные поля;
- базовая валидация регистрационной формы;
- логика завершения регистрации.

### 3.2 Вход в аккаунт

- успешный вход;
- вход с неверными данными;
- поведение системы после авторизации;
- сохранение авторизованного состояния.

### 3.3 Каталог товаров

- открытие списка товаров;
- базовая целостность отображения;
- переход к карточке товара.

### 3.4 Карточка товара

- открытие карточки товара;
- наличие базовой информации;
- возможность добавления товара в корзину.

### 3.5 Корзина

- добавление товара в корзину;
- отображение товара в корзине;
- базовая целостность сценария перехода к корзине;
- проверка поведения неавторизованного пользователя перед checkout.

---

## 4. Что было подготовлено в рамках проекта

В рамках проекта были подготовлены следующие QA-артефакты:

- описание системы;
- scope и карта рисков;
- checklist первой проверки;
- набор test cases;
- bug reports;
- evidence в формате видео;
- заметки первой QA-сессии.

---

## 5. Найденные дефекты

В ходе проверки были зафиксированы следующие дефекты:

- BUG-001. Поле First Name принимает числовые значения
- BUG-002. Поле Last Name принимает числовые и специальные символы без явной валидации
- BUG-003. Поле Mobile Number принимает буквенные символы
- BUG-004. Удаление аккаунта выполняется без шага подтверждения действия

---

## 6. Дополнительные наблюдения

Кроме подтверждённых дефектов были зафиксированы product / usability observations:

- в форме регистрации отсутствует возможность показать введённый пароль;
- отсутствует поле подтверждения пароля;
- для изменения количества товара требуется переход в карточку товара;
- в корзине отсутствует очевидная возможность уменьшить количество товара без полного удаления позиции.

Эти наблюдения не были автоматически классифицированы как дефекты, так как требуют дополнительной оценки с точки зрения требований и продуктовых решений.

---

## 7. Что пока не проверено глубоко

На текущем этапе не были глубоко проверены:

- полный сценарий checkout;
- платёжные механики;
- email-уведомления;
- работа API;
- данные в базе;
- нефункциональные проверки;
- сценарии с несколькими ролями / несколькими аккаунтами;
- пограничные сценарии корзины и заказа.

---

## 8. Основные риски по итогам проверки

По результатам первой QA-сессии можно выделить следующие риски:

### 8.1 Риски качества данных

Регистрационная форма допускает сохранение явно некорректных пользовательских данных, что указывает на слабую валидацию.

### 8.2 Риски пользовательской безопасности

Удаление аккаунта без подтверждения создаёт риск случайного разрушительного действия.

### 8.3 Риски дальнейших бизнес-сценариев

Поскольку глубоко не исследованы checkout, оплата и пост-регистрационные потоки, в этих областях остаётся высокий уровень неизвестности.

---

## 9. Общая оценка текущего состояния

По итогам первой проверки основной пользовательский путь в системе в целом работоспособен:

- пользователь может зарегистрироваться;
- пользователь может войти в систему;
- товары отображаются;
- товар можно добавить в корзину.

При этом обнаружены заметные слабые места:

- недостаточная валидация данных в регистрационной форме;
- отсутствие подтверждения для опасного действия удаления аккаунта;
- наличие ряда спорных usability-решений, требующих дополнительной оценки.

---

## 10. Вывод QA

Первая QA-сессия позволила подтвердить базовую работоспособность ключевых пользовательских сценариев и выявить начальный набор дефектов, пригодных для дальнейшего анализа и оформления в портфолио.

Проект на текущем этапе уже демонстрирует:

- базовый анализ требований и рисков;
- умение строить checklist;
- умение оформлять test cases;
- умение фиксировать bug reports;
- умение собирать evidence;
- умение отделять дефекты от product / usability observations.

Следующим логичным этапом развития проекта будет:

- доработка test design;
- оформление проекта в репозиторий / портфолио-структуру;
- при необходимости перенос части дефектов и test cases в Jira / TMS.
