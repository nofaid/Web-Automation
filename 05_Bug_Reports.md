# Bug Reports

## BUG-001. Поле First Name принимает числовые значения

### Тип

Валидация / Data Validation

### Приоритет

Средний (Medium)

### Серьёзность

Средняя (Medium)

### Предусловия

- Пользователь находится на странице регистрации аккаунта.
- Открыта форма создания нового аккаунта.

### Шаги воспроизведения

1. Открыть страницу регистрации.
2. Начать создание нового аккаунта.
3. В поле `First Name` ввести числовое значение, например `12345`.
4. Заполнить остальные обязательные поля валидными данными.
5. Завершить регистрацию.

### Ожидаемый результат

Поле `First Name` должно валидировать ввод и не допускать сохранение явно некорректного имени, состоящего только из цифр.

### Фактический результат

Система принимает числовое значение в поле `First Name` и позволяет завершить регистрацию без явной ошибки валидации.

### Риск / влияние

- в систему попадают некорректные пользовательские данные;
- снижается качество данных аккаунта;
- возможны проблемы в дальнейших бизнес-процессах, где имя пользователя используется как осмысленный атрибут;
- это указывает на слабую валидацию регистрационной формы.

### Evidence

- видео BUG-001, BUG-002, BUG-003

## BUG-002. Поле Last Name принимает числовые и специальные символы без явной валидации

### Тип

Валидация / Data Validation

### Приоритет

Средний (Medium)

### Серьёзность

Средняя (Medium)

### Предусловия

- Пользователь находится на странице регистрации аккаунта.
- Открыта форма создания нового аккаунта.

### Шаги воспроизведения

1. Открыть страницу регистрации.
2. Начать создание нового аккаунта.
3. В поле `Last Name` ввести значение с цифрами и / или специальными символами, например `@@@123`.
4. Заполнить остальные обязательные поля валидными данными.
5. Завершить регистрацию.

### Ожидаемый результат

Поле `Last Name` должно отклонять явно некорректные значения, не похожие на фамилию, либо валидировать допустимый набор символов.

### Фактический результат

Система принимает значение с цифрами и специальными символами в поле `Last Name` и позволяет завершить регистрацию без явной ошибки.

### Риск / влияние

- в систему попадают некорректные персональные данные;
- ухудшается качество данных аккаунтов;
- это может влиять на поиск, фильтрацию, отображение и иные процессы, где используются данные пользователя;
- дефект указывает на отсутствие базовой логической валидации.

### Evidence

- видео BUG-001, BUG-002, BUG-003

## BUG-003. Поле Mobile Number принимает буквенные символы

### Тип

Валидация / Data Validation

### Приоритет

Высокий (High)

### Серьёзность

Средняя (Medium)

### Предусловия

- Пользователь находится на странице регистрации аккаунта.
- Открыта форма создания нового аккаунта.

### Шаги воспроизведения

1. Открыть страницу регистрации.
2. Начать создание нового аккаунта.
3. В поле `Mobile Number` ввести буквенное значение, например `abcdef`.
4. Заполнить остальные обязательные поля валидными данными.
5. Завершить регистрацию.

### Ожидаемый результат

Поле `Mobile Number` должно принимать только значения, соответствующие формату телефонного номера, и отклонять буквенные символы.

### Фактический результат

Система принимает буквенные символы в поле `Mobile Number` и позволяет завершить регистрацию без явной ошибки валидации.

### Риск / влияние

- в систему попадает некорректный телефонный номер;
- контактные данные пользователя становятся недостоверными;
- это может повлиять на связь с пользователем, восстановление доступа и другие бизнес-процессы;
- дефект указывает на слабую проверку критичного контактного поля.

### Evidence

- видео BUG-001, BUG-002, BUG-003

## BUG-004. Удаление аккаунта выполняется без шага подтверждения действия

### Тип

Usability / Safety / Dangerous Action

### Приоритет

Высокий (High)

### Серьёзность

Высокая (High)

### Предусловия

- Пользователь зарегистрирован и авторизован в системе.

### Шаги воспроизведения

1. Войти в аккаунт.
2. Найти действие `Delete Account`.
3. Нажать кнопку удаления аккаунта один раз.

### Ожидаемый результат

Перед удалением аккаунта система должна запрашивать подтверждение действия, например через modal / confirmation dialog, поскольку удаление аккаунта является разрушительным и трудно обратимым действием.

### Фактический результат

Аккаунт удаляется сразу после одного нажатия без дополнительного подтверждения действия.

### Риск / влияние

- высокий риск случайного удаления аккаунта по ошибочному клику;
- пользователь может потерять доступ к данным без дополнительной защиты;
- поведение небезопасно для разрушительного действия;
- снижает надёжность пользовательского сценария и доверие к системе.

### Evidence

- видео BUG-004
